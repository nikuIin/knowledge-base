Уметь работать с `git` – хороший навык. Но более важно уметь с помощью него лучше описывать проект. Как описывать? С помощью `title` коммита и его `description`.

Есть несколько хороших правил для написания хорошего коммита:

1. Все коммиты (заголовки) должны быть в одном формате. Либо все со строчной буквы, либо с заглавной. 
2. Указывать префикс работы (`feat`, `fix`, etc.)
3. Указывать тег задачи (тег `Jira`)
4. Коммиты писать в повелительном наклонении (`Update CSS file`). Это как отдавать команду кому-то.

Вот структура хорошего коммита:
`fix: update CSS file (#4523)`

Теги, которые вы можете использовать:
* `fix` – разного рода исправления ошибок
* `feat` – новая фича/новая реализация или поведение
* `docs` – при внесения изменений в документацию
* `chore` – при изменении, затрагивающие инструменты разработки, такие как `git`
* `test` – для внедрение или модификации тестов

Я бы также добавил еще тег:
* `merge` – почему? Чтобы коммиты слияния не выбивались из формата.
* `refactor` – при изменении логики изменений (на самом деле лучше подумать, стоит ли его вводить, потому что ситуации, где он может использоваться, это либо `fix`, либо `feat`)
* `test-fix` — при фиксировании тестов
* `alembic` — для коммитов миграций

Также вы можете добавлять документацию, помните, главное, чтобы поняли другие, что происходит в коммите. 
`title`, `description` и абзацы в `description` нужно отделять друг от друга `\n` (пустой строкой).
