`Redux` — это технология, помогающая управлять состоянием приложения. Такие как список товаров, разные состояние счетчиком, пагинации и все, что является динамичным в проекте и влечет за собой **перезагрузку интерфейса**.

Главное столпы `Redux`:
	`store`  — единое место хранения всех динамичных данных, которые влияют на `frontend` приложение
	`action` — объект действия, который описывает, что происходит. Например `SET_USER`, `CLEAR_USER`, `INCREMENT`, `DECREMENT`. `action`-s позволяют буквально описывать, что произойдет с состоянием. 
	`reduser` — объект, которые принимает действие (optional: `payload`) и вызывает ==чистую== функцию к этому действию. То есть именно `reducer` изменяет состояние `store`. А точнее, не изменяет, а **заменяет**. В `store` должны храниться **обязательно немутабельные** данные. 
	`dispatch` — утилита, хук, которая отправляет в `reducer` определенный `action`, если `action` и функция этого требует, то данные для работы.  

---

Важно понимать, **когда стоит использовать `Redux`**.  

`Redux` стоит использовать **только тогда, когда нам нужно на уровне сессии**, то есть **без перезагрузки страницы**, сохранять состояние.  

### Основные моменты использования `Redux`:
- **Состояние приложения** должно влиять на поведение или отображение UI и UX компонента.
- При **перезагрузке страницы браузер обновляет весь JavaScript код**.
- `Redux` хранит состояние в самом JavaScript коде, поэтому при перезагрузке:
  - Всё вернётся в изначальное состояние
  - Состояние в `Redux` будет потеряно

### Если важно сохранять значения после перезагрузки:
- Нужно реализовать собственное решение для сохранения, например:
  - Использовать `Local Storage`
  - Использовать библиотеку `Redux Persistence`

Таким образом, `Redux` идеально подходит для **управления состоянием, которое нужно хранить только в рамках одной сессии без перезагрузок**. Для долговременного сохранения состояния требуется дополнительная реализация.

## Отличие `Redux` от `Context`

 Также стоит помнить, как указано в статье, закреплённой в начале, что **`Redux`** — это инструмент для *управления состоянием* приложения. Он позволяет нам легко описать состояние нашего приложения в **чистых функциях** — **`редьюсерах`** — и аккуратно изменять его.

Мы можем добиться похожего результата с помощью следующих инструментов:
- **`useContext`**
- **`useState`**

Да, используя просто *контекст*, можно управлять состоянием, например:
- Подписывать данные на **`localStorage`** или **`sessionStorage`**
- Использовать хуки

Однако нужно помнить, что **`Context`** предназначен **только для передачи** состояния. Он **не предоставляет автоматического решения** для динамического изменения состояния без дополнительных хуков и механизмов, таких как:
- **`useContext`**
- Хранение данных в **`localStorage`** и т.д.

Другими словами, контекст **просто передаёт состояние**, а **`Redux`** уже позволяет описать работу с этим состоянием, обеспечивая **динамическое управление**.
